# feat: AI-Powered i18n — Vercel AI SDK Translation, Locale Variants, Admin Locale Management

## Summary

This plan adds opt-in AI-powered internationalization to `@hono-cms`. When a collection declares `localization: true`, the CMS generates a companion locale-variants table, exposes `?locale=` query semantics on content routes, and enqueues background translation jobs via QStash. The Vercel AI SDK abstracts the translation provider — Anthropic, OpenAI, or Vercel AI Gateway — behind a uniform interface. Locale variants carry a `translatedBy` flag that distinguishes AI drafts from human-reviewed translations, giving editors a clear review workflow. The admin SPA gains a locale switcher, per-locale status badges, and a bulk-backfill tool for adding locales after initial content exists.

This is Plan 013 of 18. It depends on the `createCMS` API and `CMSConfig` type surface (Plan 002), the `DatabaseAdapter` interface (Plan 003), the content routes framework (referenced as Plan 005), the admin SPA foundation (referenced as Plan 007), the QStash jobs layer (referenced as Plan 010), and the SDK type generator (Plan 017). Every locale-specific behavior in this plan is gated by `i18n` being defined in `CMSConfig` — the feature is entirely tree-shaken when absent.

## Problem Frame

Enterprise CMSes (Strapi, Contentful) treat localization as a premium feature. Contentful charges per locale per space. Strapi's i18n plugin requires human translation workflows: export JSON, send to a translation agency, import back — a cycle that takes days. For a CMS targeting small projects, this friction is a non-starter.

The addressable shift: use the AI SDK as a first-class translation provider. Claude Haiku 4.5 and GPT-4o Mini can translate a 1,000-word article for under $0.01. AI translation quality on standard editorial content (blog posts, product descriptions, marketing copy) is human-quality for most Western European language pairs. The result: an editor saves in English, and Spanish, French, and German drafts appear in the admin within seconds — ready for review, not from scratch.

The key design problem is data model: where do locale variants live relative to the canonical row? The wrong answer (a locale column on the main table making every row locale-aware) pollutes queries for monolingual content and creates a non-obvious primary key. The right answer (a separate `{collection}_locale_variants` table) is clean, composable, and does not touch the main table schema for collections that do not opt in.

## Key Technical Decisions

## Research Insights

**Best Practices:**
- Store source revision, locale revision, provider, model, and prompt metadata so translation writes are version-aware.
- Decide explicitly whether publish/review state is per document or per locale variant before implementation proceeds.
- Keep locale fallback rules aligned across API responses, admin UI, and generated SDK helpers.

**Security Considerations:**
- Default to not serving AI-generated translations publicly until reviewed, or require explicit opt-in per collection/workflow.
- Never send RBAC-hidden fields, internal notes, private drafts, or restricted collections to AI providers automatically.

**Edge Cases:**
- Translation jobs should not overwrite human-edited or human-reviewed locale variants without an explicit policy.
- Locale status endpoints should not leak workflow state for unreadable or draft content.

### KTD-1: Separate `{collection}_locale_variants` table (not a locale column on the main table)

**Decision:** Localized fields are stored in a separate `{collection}_locale_variants` table. The main `articles` table retains only the default-locale content and all non-localizable fields (relations, media, dates, booleans, numbers). Locale variants are rows in `articles_locale_variants` with a `document_id` FK to `articles.id`.

**Rejected alternative — locale column on main table (Option A):** Adding `locale TEXT NOT NULL DEFAULT 'en'` and `document_id TEXT` to the main table makes every row locale-aware. This forces every query — even from sites that never use i18n — to include a `WHERE locale = ?` filter to avoid returning duplicate rows. It also complicates the primary key (must be composite on `id` + `locale`, or use a separate `document_id` as the stable canonical identifier). For collections with `localization: false`, this adds dead columns. The ideation doc floats this as the approach ("same pattern as `status` for draft/publish") but the actual constraint is different: `status` adds one column to one existing row; locale would multiply rows in the same table, requiring a two-level key scheme.

**Why the separate table wins:**
- The main table schema is unchanged for non-localized collections — zero overhead
- Localizable fields are cleanly separated from shared fields in the physical schema
- Queries for the default locale (most API traffic) hit the main table only — the join is optional
- Adding a locale later (bulk backfill) is an insert-only operation, not a schema migration on the main table
- The `documents` + `locale_variants` split mirrors Strapi v5's Document Service model and Contentful's entry/locale architecture — both validated at scale

**Schema shape (directional):**

```
articles              (main table — default locale content + shared fields)
  id TEXT PK
  title TEXT           ← default locale
  body TEXT            ← default locale
  author_id TEXT       ← shared, not localized
  cover_image_id TEXT  ← shared, not localized
  published_at TIMESTAMP ← shared
  created_at TIMESTAMP
  updated_at TIMESTAMP

articles_locale_variants  (companion table — one row per locale per document)
  id TEXT PK
  document_id TEXT FK → articles.id
  locale TEXT          ← BCP 47 code: 'es', 'es-MX', 'fr', 'de'
  title TEXT
  body TEXT
  translated_by TEXT   ← 'ai' | 'human' | 'pending'
  translation_status TEXT ← 'pending' | 'in_progress' | 'complete' | 'error'
  translation_error TEXT   ← nullable, AI error message
  translated_at TIMESTAMP
  created_at TIMESTAMP
  updated_at TIMESTAMP
  UNIQUE(document_id, locale)
```

The `UNIQUE(document_id, locale)` constraint prevents duplicate locale variants and enables upsert semantics when retranslating.

**Localizable field classification:** At schema definition time, fields of type `string`, `text`, and `richtext` are localizable by default. Fields of type `relation`, `media`, `date`, `datetime`, `boolean`, `integer`, `float`, `json` are shared — they live only on the main table and are not duplicated in locale variants. This classification is inspectable at the `CollectionField` type level and drives both schema generation and translation job field selection.

### KTD-2: Translate in background after save (lazy via QStash)

**Decision:** Translation is always asynchronous. When an editor saves a document in the default locale, the CMS immediately responds with the saved document and concurrently enqueues one QStash job per non-default locale. The locale variants move through `pending → in_progress → complete` states asynchronously. The admin SPA polls `/api/{collection}/{id}/locales` or receives a real-time update (Plan 007's WebSocket or SSE channel) to reflect status.

**Rejected alternative — eager (synchronous) translation:** Translating all locales synchronously before responding to the save request would block the editor for several seconds (3 locales × ~1-2s per AI call = 3-6s response time). This is unacceptable UX and would also consume the entire request budget of serverless functions (Cloudflare Workers: 30s CPU wall time; Vercel Edge: 25s). Even a single translation call risks timeout on large rich-text documents.

**`translateOnPublish` flag (see U2):** The `i18n.translateOnPublish` config option modifies timing within the async path: when `true`, jobs are only enqueued when the document transitions to `published` status, not on every save. This is useful for draft-heavy workflows where editors iterate many times before publishing. The implementation uses the content lifecycle hooks (Plan 005) to check publish state before enqueuing.

### KTD-3: Richtext and markdown preservation via structured prompt engineering

**Decision:** The translation prompt wraps rich-text content in explicit delimiters and instructs the model to return only the translated content with no prose commentary, preserving all HTML tags, markdown syntax, Slate.js JSON structure, and link href/src values verbatim.

Rich-text content in the CMS is stored as one of: raw HTML, Markdown string, or Slate.js JSON (depending on the richtext field's `format` sub-option). The translation job detects the format and applies the appropriate prompt:

- **HTML format:** `"Translate the following HTML from {source} to {target}. Return ONLY the translated HTML. Preserve all HTML tags, attributes, href values, src values, class names, and id attributes exactly. Do not translate content inside <code>, <pre>, or <script> tags. Do not add any surrounding text or explanation.\n\n{content}"`
- **Markdown format:** Same instruction set, emphasizing backtick blocks, inline code spans, link URLs, and image paths as untouchable.
- **Slate JSON format:** Extract only `text` leaf nodes (those not marked `code: true`) for translation, reassemble the Slate tree with translated text nodes. The model never sees the full JSON — only the extracted strings are translated, preserving all formatting marks.

This structured approach is more reliable than asking the model to "preserve formatting" in a vague instruction, because it separates the translation surface from the structural surface.

### KTD-4: AI-translated content is `translatedBy: 'ai'` and requires explicit review before being treated as authoritative

**Decision:** AI-translated locale variants are stored with `translatedBy: 'ai'` and are served via the API to end users immediately (they are not "draft-only"). However, the admin SPA prominently labels them "AI draft — review before publishing" and the `translatedBy` field is surfaced in API responses. Whether to show AI-translated content on the public site is the operator's choice — the CMS provides the data; the frontend decides trust.

**Rationale:** Blocking AI content from the public API until reviewed would make i18n useless for small projects where the operator trusts AI quality for their language pairs. Automatically publishing without any signal would mislead editors about content provenance. The `translatedBy` field in the API response lets frontend code add its own trust signal if desired (e.g., "This content was auto-translated").

A future `requireReviewBeforePublish: true` config flag can change this behavior (block serving `translatedBy: 'ai'` variants until `translatedBy: 'human'`), but is deferred to v2.

### KTD-5: Bulk backfill strategy for newly added locales

**Decision:** When a new locale is added to `i18n.locales` after content already exists, the operator runs a backfill via the admin UI (`/settings/i18n` → "Translate all to [locale]") or the CLI (`cms i18n backfill --locale es`). The backfill enqueues one QStash job per existing document in the collection. There is no automatic backfill on server boot — triggering translation for potentially thousands of documents silently on startup would be an operational surprise and cost shock.

The backfill API endpoint (`POST /cms/admin/i18n/backfill`) accepts `{ locale, collection? }` and returns immediately with a job count. QStash handles concurrency naturally via its queue (jobs are processed one at a time per queue name, preventing 50 parallel AI calls). The admin UI shows a progress bar via polling `/cms/admin/i18n/backfill/status`.

### Translation lifecycle

```
Editor saves article (default locale 'en')
  │
  ├─ Content route handler: INSERT INTO articles (...)
  │
  ├─ i18nMiddleware detects: collection has localization: true
  │    AND config.i18n exists AND autoTranslate: true
  │
  ├─ For each locale in (config.i18n.locales - default):
  │    qstash.publish({
  │      url: '/cms/jobs/translation',
  │      body: { documentId, collection: 'articles', targetLocale: 'es' },
  │      retries: 0    ← no auto-retry; error stored on variant row
  │    })
  │    INSERT INTO articles_locale_variants (document_id, locale, translation_status)
  │    VALUES (?, 'es', 'pending')
  │    ON CONFLICT DO UPDATE SET translation_status = 'pending'
  │
  └─ Response returns immediately (document saved; locales enqueued)

QStash delivers POST /cms/jobs/translation { documentId, collection, targetLocale }
  │
  ├─ Verify QStash signature (QSTASH_CURRENT_SIGNING_KEY)
  │
  ├─ UPDATE articles_locale_variants SET translation_status = 'in_progress'
  │
  ├─ SELECT * FROM articles WHERE id = documentId  ← load source document
  │
  ├─ Identify localizable fields: [title (string), body (richtext)]
  │
  ├─ For each localizable field:
  │    aiModel = createAIProvider(config.i18n)
  │    translatedText = generateText(aiModel, buildPrompt(field, 'en', 'es', value))
  │
  ├─ UPSERT articles_locale_variants SET
  │    title = translatedTitle,
  │    body = translatedBody,
  │    translated_by = 'ai',
  │    translation_status = 'complete',
  │    translated_at = now()
  │
  └─ On AI error:
       UPDATE articles_locale_variants SET
         translation_status = 'error',
         translation_error = errorMessage
```

### Locale query fallback chain

```
GET /api/articles?locale=es-MX

  1. SELECT * FROM articles_locale_variants
     WHERE document_id = ? AND locale = 'es-MX'
     → found? return it

  2. Not found. Fallback enabled?
     locale = stripSubtag('es-MX') → 'es'
     SELECT * FROM articles_locale_variants
     WHERE document_id = ? AND locale = 'es'
     → found? return it with X-Locale-Fallback: es header

  3. Not found. Return default locale from main table
     SELECT * FROM articles WHERE id = ?
     → return with X-Locale-Fallback: en header

  4. &fallback=false? → 404 at step 2
```

### Provider abstraction

```
createAIProvider(config: I18nConfig) → LanguageModelV1

  provider: 'anthropic'  → anthropic(config.model)     (@ai-sdk/anthropic)
  provider: 'openai'     → openai(config.model)         (@ai-sdk/openai)
  provider: 'gateway'    → createGateway()(config.model) (ai package gateway)
```

## Implementation Units

### U1. Locale Data Model and Drizzle Schema Generator Extension

**Goal:** When a collection declares `localization: true`, the Drizzle schema generator emits a `{collection}_locale_variants` companion table containing all localizable fields plus locale metadata columns. The main table is unchanged except for not duplicating shared fields.

**Requirements:**
- Schema generator detects `localization: true` on `CollectionConfig`
- Emits companion table with `document_id FK`, `locale TEXT`, all `string`/`text`/`richtext` fields cloned from the main table, `translated_by`, `translation_status`, `translation_error`, `translated_at`, `created_at`, `updated_at`
- `UNIQUE(document_id, locale)` constraint on companion table
- Localizable field classification is defined at `CollectionField` type level: `string`, `text`, `richtext` → localizable; all others → shared
- `?locale=` filtering: when locale param present, LEFT JOIN or subquery against locale_variants table; when absent, query main table only (no performance penalty for non-i18n paths)
- Fallback chain: `es-MX → es → default` via two-step query (subtag strip, then default)
- `X-Locale-Fallback: {actualLocale}` response header on fallback

**Dependencies:** Plan 003 (`DatabaseAdapter`), Plan 002 (`CollectionConfig`, `CollectionField` types)

**Files:**
- `packages/core/src/i18n/schema-gen.ts` — locale variant table generator
- `packages/core/src/i18n/locale-query.ts` — locale-aware query builder and fallback logic
- `packages/core/src/i18n/bcp47.ts` — subtag strip utility, BCP 47 format validation
- `packages/schema/src/collection.ts` — extend `CollectionField` with `localizable: boolean` classification (derived from field type, not user-set)
- `packages/core/src/i18n/__tests__/schema-gen.test.ts`
- `packages/core/src/i18n/__tests__/locale-query.test.ts`

**Approach:**

The schema generator is a pure function: `generateLocaleVariantsTable(collection: CollectionConfig, dialect: 'sqlite' | 'pg' | 'mysql') → DrizzleTableDefinition`. It is called by the main schema compiler when `collection.localization === true`. The output is appended to the same schema file as the main table.

Localizable field classification lives in a utility `isLocalizableField(field: CollectionField): boolean` that returns `true` for `string`, `text`, `richtext` field types and `false` for everything else. This function is the single authority — used by schema generation AND by the translation job (U3) to identify which fields to translate.

Locale-aware queries use a conditional JOIN pattern. When `locale` param is absent, the query is `SELECT * FROM articles` — no join, no overhead. When `locale` param is present:

```
SELECT
  a.*,                            ← shared fields from main table
  COALESCE(lv.title, a.title) as title,   ← locale variant or fallback
  COALESCE(lv.body, a.body) as body,
  lv.translated_by,
  lv.translation_status
FROM articles a
LEFT JOIN articles_locale_variants lv
  ON lv.document_id = a.id AND lv.locale = ?
```

This pattern (COALESCE with main table as fallback) handles the default locale case correctly: if no locale variant exists (either fallback disabled or variant not yet created), the main table values come through.

Fallback chain is implemented as:
1. Try exact locale match (`es-MX`)
2. If no match and locale has subtag: strip subtag, try base language (`es`)
3. If no match: return main table row (default locale content) with fallback header
4. If `fallback=false`: return 404 at step 2

The `bcp47.ts` utility validates locale codes at config time (using a regex against the BCP 47 grammar, not a full IANA registry lookup — sufficient for the common use case) and provides the subtag strip function.

**Technical design (directional):**

```
generateLocaleVariantsTable(collection, dialect):
  localizableFields = collection.fields.filter(isLocalizableField)
  columns = [
    id: text().primaryKey(),
    documentId: text().notNull().references(() => mainTable.id, { onDelete: 'cascade' }),
    locale: text().notNull(),
    ...cloneColumns(localizableFields, dialect),
    translatedBy: text().default('pending'),   // 'ai' | 'human' | 'pending'
    translationStatus: text().default('pending'),
    translationError: text(),
    translatedAt: timestamp(),
    createdAt: timestamp().defaultNow(),
    updatedAt: timestamp().defaultNow(),
  ]
  table = createTable(`${collection.name}_locale_variants`, columns)
  table.uniqueIndex(['documentId', 'locale'])
  return table
```

**Patterns to follow:** The draft/publish schema pattern from Plan 005 (adds `status` and `published_at` to the main table) is the reference for how the schema generator extends existing collections. The locale variant approach follows the same "generator function called by main compiler" pattern.

**Test scenarios:**
- Schema generator emits a `_locale_variants` table when `localization: true`; no `_locale_variants` table when `localization: false`
- Emitted table contains only `string`/`text`/`richtext` fields from the source collection, not `relation`, `media`, or `boolean` fields
- `UNIQUE(document_id, locale)` index is present in the emitted schema
- `document_id` FK has `ON DELETE CASCADE` so variant rows are cleaned up when the main row is deleted
- `?locale=es` query returns Spanish variant when it exists (COALESCE returns lv.title, not a.title)
- `?locale=es` query returns main table content with `X-Locale-Fallback: en` header when no Spanish variant exists
- `?locale=es-MX` subtag fallback: tries `es-MX` → not found, tries `es` → found, returns with `X-Locale-Fallback: es` header
- `?locale=es&fallback=false` returns 404 when Spanish variant does not exist
- BCP 47 validator rejects `'english'`, `'EN'` (wrong case), `'en_US'` (underscore), accepts `'en'`, `'es-MX'`, `'zh-Hans'`, `'pt-BR'`
- `isLocalizableField` returns `true` for `string`, `text`, `richtext`; `false` for `relation`, `media`, `boolean`, `date`, `integer`, `float`, `json`

**Verification:** Schema generator unit tests pass. A test collection with mixed field types produces a locale_variants table containing only the localizable fields. Integration test: insert a document, insert a locale variant, query with `?locale=es` returns the variant values; delete the document, confirm variant row is cascade-deleted.

### U2. `I18nConfig` TypeScript Type

**Goal:** Define the complete TypeScript type for the `i18n` key in `CMSConfig`, including provider discriminated union, BCP 47 locale array, default locale constraint, and optional behavior flags.

**Requirements:**
- `I18nConfig` is optional on `CMSConfig` — omitting it entirely disables i18n with zero runtime overhead
- Provider discriminated union: `'anthropic'`, `'openai'`, `'gateway'`
- Each provider variant constrains the valid `model` string (string literal union for known models, fallback to `string & {}` for extensibility)
- `locales: [string, ...string[]]` — non-empty array of BCP 47 codes
- `default: string` — must be a member of `locales` (enforced at runtime with a clear error, not at type level — the TypeScript constraint would require template literal magic that hurts inference)
- `autoTranslate?: boolean` — default `true`; when `false`, translation is never enqueued automatically (only manual trigger via API)
- `translateOnPublish?: boolean` — default `false`; when `true`, enqueue only when document status transitions to `published`

**Dependencies:** Plan 002 (`CMSConfig`)

**Files:**
- `packages/core/src/i18n/config.ts` — `I18nConfig`, `AnthropicI18nConfig`, `OpenAIi18nConfig`, `GatewayI18nConfig` types
- `packages/core/src/i18n/bcp47.ts` — `validateLocaleCode(code: string): boolean`, `validateI18nConfig(config: I18nConfig): void` (throws on invalid default locale or empty locales array)
- `packages/schema/src/config.ts` — extend `CMSConfig` with optional `i18n?: I18nConfig`
- `packages/core/src/i18n/__tests__/config.test.ts`

**Approach:**

Three discriminated variants share a common base:

```
BaseI18nConfig = {
  locales: [string, ...string[]]
  default: string
  autoTranslate?: boolean       // default: true
  translateOnPublish?: boolean  // default: false
}

AnthropicI18nConfig = BaseI18nConfig & {
  provider: 'anthropic'
  model: 'claude-haiku-4-5' | 'claude-3-5-sonnet-20241022' | (string & {})
}

OpenAII18nConfig = BaseI18nConfig & {
  provider: 'openai'
  model: 'gpt-4o-mini' | 'gpt-4o' | (string & {})
}

GatewayI18nConfig = BaseI18nConfig & {
  provider: 'gateway'
  model: string   // 'anthropic/claude-haiku-4-5' format
  gatewayId?: string  // Vercel AI Gateway ID, if not using default
}

I18nConfig = AnthropicI18nConfig | OpenAII18nConfig | GatewayI18nConfig
```

The `(string & {})` trick preserves autocomplete for the known model names while accepting any string — this is the standard Vercel AI SDK pattern for model names.

`validateI18nConfig` runs inside `createCMS` during bootstrap (synchronous) and throws a descriptive `CMSConfigError` if:
- `locales` is empty
- `default` is not in `locales`
- Any locale code fails BCP 47 basic format validation

**Patterns to follow:** The `db` provider discriminated union in Plan 002/003 (`SqliteDbConfig | D1DbConfig | PostgresDbConfig`) is the exact pattern to follow for `AnthropicI18nConfig | OpenAII18nConfig | GatewayI18nConfig`.

**Test scenarios:**
- `{ provider: 'anthropic', model: 'claude-haiku-4-5', locales: ['en', 'es'], default: 'en' }` — valid, passes validation
- `{ locales: ['en', 'es'], default: 'fr' }` — throws `CMSConfigError: default locale 'fr' is not in locales array`
- `{ locales: [], default: 'en' }` — throws `CMSConfigError: locales array must be non-empty`
- `{ locales: ['en', 'EN'] }` — BCP 47 validator catches `'EN'` as invalid (must be lowercase)
- TypeScript: `provider: 'anthropic'` with `model: 'claude-haiku-4-5'` satisfies `I18nConfig`; `provider: 'unknown'` is a TypeScript error
- When `i18n` key is omitted from `CMSConfig`, no i18n middleware is mounted and no locale routes are registered

### U3. Translation Job Handler

**Goal:** Implement `translationJob`, the QStash-delivered background handler that loads a source document, identifies and translates localizable fields via the Vercel AI SDK, and writes the result as a locale variant row.

**Requirements:**
- Handler signature: `translationJob(payload: TranslationJobPayload, config: I18nConfig, db: DatabaseAdapter): Promise<void>`
- `TranslationJobPayload`: `{ documentId: string; collection: string; targetLocale: string; sourceLocale: string }`
- Loads source document from main table via `db`
- Identifies localizable fields using `isLocalizableField` (U1)
- Translates each localizable field individually (not as a single batch) — isolates failures per field
- Handles three richtext formats: raw HTML, Markdown, Slate JSON (detected from field `format` sub-option)
- Slate JSON: extract text leaf nodes → translate → reassemble (model never sees full JSON)
- Writes result via UPSERT into `{collection}_locale_variants` with `translatedBy: 'ai'`, `translationStatus: 'complete'`, `translatedAt: now()`
- Error handling: any AI provider error → set `translationStatus: 'error'`, `translationError: error.message`; do not throw (QStash would retry — we do not want retries)
- Updates `translationStatus: 'in_progress'` before starting AI calls so admin UI reflects active job

**Dependencies:** U1 (schema + `isLocalizableField`), U2 (`I18nConfig`), U4 (`createAIProvider`)

**Files:**
- `packages/core/src/jobs/translation.ts` — `translationJob` handler, `buildTranslationPrompt`, `extractSlateTextNodes`, `reassembleSlateTree`
- `packages/core/src/jobs/translation.types.ts` — `TranslationJobPayload`, `TranslationJobResult`
- `packages/core/src/jobs/__tests__/translation.test.ts`

**Approach:**

The handler is a pure async function with no HTTP coupling — it is called by the Hono route handler at `POST /cms/jobs/translation` after QStash signature verification (U5). This separation allows unit testing the translation logic without mocking HTTP.

Field-by-field translation loop (directional):

```
for each field in localizableFields:
  if field.type === 'richtext':
    if field.format === 'slate':
      nodes = extractSlateTextNodes(rawValue)
      translatedNodes = await translateTextNodes(nodes, model, sourceLocale, targetLocale)
      translatedValue = reassembleSlateTree(rawValue, translatedNodes)
    else:
      // html or markdown — translate directly with format-specific prompt
      translatedValue = await generateText(model, buildPrompt(field.type, field.format, sourceLocale, targetLocale, rawValue))
  else:
    // string or text
    translatedValue = await generateText(model, buildSimplePrompt(sourceLocale, targetLocale, rawValue))
  results[field.name] = translatedValue
```

Prompt templates are constants in the module, not constructed ad-hoc, so they are easy to audit and adjust. The HTML prompt explicitly lists untranslatable attributes. The Markdown prompt explicitly names code fences, inline code, and link syntax. See KTD-3 for the full prompt strategy.

Error handling: wrap the entire field-translation loop in try/catch. On any error, immediately UPSERT the error state and return (do not throw). Log the error with enough context (documentId, collection, targetLocale, errorMessage) for operator debugging.

**Patterns to follow:** The QStash job handler pattern from Plan 010 (crons/jobs) is the reference — same signature verification step, same no-throw design for background jobs, same structured error storage.

**Test scenarios:**
- Happy path: document with `title` (string) and `body` (richtext/html) is loaded; both fields are translated; locale variant row is upserted with `translation_status: 'complete'`, `translated_by: 'ai'`
- Richtext HTML preservation: HTML input `<h1>Hello <strong>world</strong></h1>` — translated output preserves `<h1>` and `<strong>` tags; only inner text is changed
- Richtext Markdown preservation: `## Hello\n\nSome [link](https://example.com) text` — URL is unchanged in translated output; heading marker preserved
- Slate JSON: only `text` leaf nodes (not `code: true` nodes) are extracted and translated; JSON structure is reassembled identically
- AI provider error: `generateText` throws a network error → `translation_status` is set to `'error'`, `translation_error` contains the error message, handler returns without throwing
- Non-localizable fields (relation, media, boolean) are NOT included in the translation payload — they are not passed to the AI
- `in_progress` status is written before AI calls begin; `complete` status is written after all fields succeed
- Documents with empty string fields: empty strings are not sent to the AI (short-circuit to empty translated value)
- Source document not found (deleted between enqueue and job delivery): handler returns early with a structured log entry, no error status (no locale variant row to update)

**Verification:** Unit tests cover all richtext formats with mocked AI SDK. Integration test (with a test AI provider returning static responses) confirms full upsert flow. A test for the error path confirms the locale variant row has `translation_status: 'error'` and the handler returns without re-throwing.

### U4. AI Provider Abstraction (`createAIProvider`)

**Goal:** Implement `createAIProvider(config: I18nConfig)` that returns a Vercel AI SDK `LanguageModelV1` instance backed by the configured provider. Provider packages (`@ai-sdk/anthropic`, `@ai-sdk/openai`) are optional peer dependencies, dynamically imported at runtime to avoid bundling unused providers.

**Requirements:**
- `createAIProvider(config: I18nConfig): LanguageModelV1`
- Dynamic import per provider: `await import('@ai-sdk/anthropic')` (only when `provider: 'anthropic'`)
- Clear error if required peer dep is not installed: `"@ai-sdk/anthropic is required when provider is 'anthropic'. Install it: bun add @ai-sdk/anthropic"`
- Gateway provider: use `createProviderRegistry` from `ai` package or direct model ID format `'anthropic/claude-haiku-4-5'`
- API key resolution: `ANTHROPIC_API_KEY` env var for Anthropic; `OPENAI_API_KEY` for OpenAI; `AI_GATEWAY_URL` + optional key for gateway
- Rate limiting: QStash's queue serialization is the primary rate-limiting mechanism (one job processed at a time per queue name). `createAIProvider` itself does not implement rate limiting — the queue design enforces it.
- Concurrency note: operators can configure multiple QStash queues for higher throughput; document this in config reference

**Dependencies:** U2 (`I18nConfig`)

**Files:**
- `packages/core/src/providers/ai.ts` — `createAIProvider` factory
- `packages/core/src/providers/ai.types.ts` — provider-specific config extension types if needed
- `packages/core/src/providers/__tests__/ai.test.ts`

**Approach:**

Dynamic import is used instead of static import so that unused provider packages are never bundled. The factory is async (returns `Promise<LanguageModelV1>`) — called once per job invocation, not at `createCMS` time, so the async cost is per-job.

Provider resolution (directional):

```
createAIProvider(config):
  switch config.provider:
    case 'anthropic':
      { anthropic } = await import('@ai-sdk/anthropic').catch(() => throw MissingPeerDepError)
      return anthropic(config.model)

    case 'openai':
      { openai } = await import('@ai-sdk/openai').catch(() => throw MissingPeerDepError)
      return openai(config.model)

    case 'gateway':
      { createOpenAI } = await import('@ai-sdk/openai')
      return createOpenAI({
        baseURL: env.AI_GATEWAY_URL ?? 'https://gateway.ai.vercel.com/v1/{team}/{gateway}',
        apiKey: env.AI_GATEWAY_API_KEY,
      })(config.model)
```

The `ai` package itself (`generateText`) is a direct dependency of `@hono-cms/core`, not a peer dep — it is always bundled. Only the provider-specific packages (`@ai-sdk/anthropic`, `@ai-sdk/openai`) are peer deps.

Rate limiting rationale: QStash queues (Plan 010) process one message at a time per queue by default. For an installation with 1,000 articles and 3 target locales, that is 3,000 jobs processed serially — each taking ~1-2s = ~1 hour total. This is acceptable. Operators who want higher throughput can configure QStash with `parallelism > 1` on the queue, accepting higher API cost and potential rate limit risk. The CMS does not need to implement its own concurrency limiter.

**Patterns to follow:** The provider registry pattern from Plan 002/003 (how database adapters register themselves) is the reference. The dynamic import pattern for optional peer deps should match the existing `createStorageAdapter` factory in `packages/core`.

**Test scenarios:**
- `createAIProvider({ provider: 'anthropic', model: 'claude-haiku-4-5', ... })` — returns an object satisfying `LanguageModelV1` interface (mock `@ai-sdk/anthropic` in tests)
- `createAIProvider({ provider: 'anthropic', ... })` when `@ai-sdk/anthropic` is not installed — throws `MissingPeerDepError` with the install command in the message
- `createAIProvider({ provider: 'openai', ... })` when `@ai-sdk/openai` is not installed — same pattern
- `createAIProvider({ provider: 'gateway', model: 'anthropic/claude-haiku-4-5', ... })` — constructs an OpenAI-compatible client pointed at the gateway URL from env
- API key missing for Anthropic provider — error includes which env var is required (`ANTHROPIC_API_KEY`)
- Two consecutive calls with the same config return valid model instances (no singleton caching required at this level — QStash jobs are stateless)

**Verification:** Unit tests with mocked dynamic imports confirm provider selection. A build-time test confirms that bundling a consumer that configures only `provider: 'anthropic'` does NOT include `@ai-sdk/openai` in the output bundle (tree-shaking check via bundle analysis).

### U5. Locale API Routes

**Goal:** Wire `?locale=` query parameter handling into existing content routes (list and detail) and add two new locale-specific endpoints: `POST /{collection}/{id}/translate` (manual trigger) and `GET /{collection}/{id}/locales` (status listing).

**Requirements:**
- `GET /api/{collection}?locale=es` — locale-aware list query; uses U1 fallback chain
- `GET /api/{collection}/{id}?locale=es` — locale-aware detail query
- `GET /api/{collection}/{id}?locale=es&fallback=false` — strict mode: 404 if variant absent
- `POST /api/{collection}/{id}/translate` body: `{ targetLocale: string }` — manually enqueue translation job for a specific locale; admin-only (RBAC check: `translate:{collection}` permission or admin role)
- `GET /api/{collection}/{id}/locales` — returns all configured locales with their variant status (`{ locale, translationStatus, translatedBy, translatedAt }` per locale); returns `not_started` for locales with no variant row
- `POST /cms/jobs/translation` — QStash callback endpoint; verifies QStash signature; calls `translationJob`; returns `200 OK` always (no throw)
- `POST /cms/admin/i18n/backfill` — bulk enqueue; accepts `{ locale: string, collection?: string }`; admin-only
- `GET /cms/admin/i18n/backfill/status` — returns count of pending/complete jobs for in-progress backfills
- All locale routes are only registered when `config.i18n` is defined

**Dependencies:** U1 (locale query builder), U3 (translation job handler), U4 (AI provider), U2 (`I18nConfig`)

**Files:**
- `packages/core/src/routes/locale-routes.ts` — `/translate`, `/locales`, `/cms/jobs/translation`, `/cms/admin/i18n/backfill` routes
- `packages/core/src/routes/content-routes.ts` — extend existing list/detail handlers to accept `locale` and `fallback` query params (modify, not new file)
- `packages/core/src/i18n/middleware.ts` — `i18nAfterSaveMiddleware` — hook that enqueues translation jobs after content save
- `packages/core/src/routes/__tests__/locale-routes.test.ts`

**Approach:**

`?locale=` is handled as a query parameter middleware layer applied to content routes when `config.i18n` is defined. The content route handler calls `resolveLocale(query.locale, query.fallback, collectionConfig, db)` which implements the U1 fallback chain and returns `{ rows, actualLocale, isFallback }`. If `isFallback` is true, the response includes `X-Locale-Fallback: {actualLocale}` header.

The `i18nAfterSaveMiddleware` runs as a Hono `after` hook on `POST /api/{collection}` and `PUT /api/{collection}/{id}`. It checks:
1. Is `config.i18n` defined?
2. Does the collection have `localization: true`?
3. Is `config.i18n.autoTranslate !== false`?
4. If `translateOnPublish: true`, is the saved document in `published` status?

If all checks pass, it enqueues jobs for each non-default locale.

The `POST /cms/jobs/translation` endpoint verifies the QStash signature using the `QSTASH_CURRENT_SIGNING_KEY` environment variable. QStash sends a `Upstash-Signature` header; the verification uses `@upstash/qstash`'s `verifySignature` function. On signature failure: return `401`. On handler success or handled error: return `200` (QStash will not retry a `200` response).

Manual translation trigger (`POST /{collection}/{id}/translate`): validates the `targetLocale` is in `config.i18n.locales`, checks it is not the default locale, upserts a `pending` variant row, enqueues the QStash job, and returns `{ status: 'enqueued', targetLocale }`.

**Patterns to follow:** The Plan 005 content routes are the base pattern. The QStash signature verification pattern from Plan 010 (cron jobs) is the reference for `POST /cms/jobs/translation`.

**Test scenarios:**
- `GET /api/articles?locale=es` with Spanish variant present — returns variant fields, no fallback header
- `GET /api/articles?locale=es` with no Spanish variant — returns main table fields, `X-Locale-Fallback: en` header
- `GET /api/articles?locale=es-MX` with only `es` variant — returns `es` variant, `X-Locale-Fallback: es` header
- `GET /api/articles?locale=es&fallback=false` with no Spanish variant — returns `404`

- `GET /api/articles/{id}/locales` — returns all configured locales; `complete` for translated, `not_started` for never-queued locales

- `POST /api/articles/{id}/translate` with `targetLocale: 'en'` (default locale) — returns `400 Bad Request: cannot translate into the default locale`
- `POST /api/articles/{id}/translate` with `targetLocale: 'xx'` (not in config locales) — returns `400: locale 'xx' is not configured`

### U7. Locale-Aware SDK Types

**Goal:** Extend the auto-generated SDK types (Plan 017) to include locale-aware query parameters and the `LocaleVariant<T>` return type, so consumers using the typed SDK client get full type safety for locale queries.

**Requirements:**
- `findMany({ locale: 'es', fallback?: boolean })` on any localized collection is correctly typed

- When no `locale` param is passed, return type is the base collection type (no locale fields)
- Non-localized collections (`localization: false`) do not accept a `locale` param — TypeScript error if passed
- `translate({ id, targetLocale })` method on localized collection clients — calls `POST /{collection}/{id}/translate`
- `getLocales({ id })` method — calls `GET /{collection}/{id}/locales`, returns `LocaleStatus[]`

**Dependencies:** U1, U2, Plan 017 (SDK type generator)

**Files:**
- `packages/core/src/types/locale.ts` — `LocaleVariant<T>`, `LocaleQueryParams`, `LocaleStatus`, `LocalizedCollectionClient<T>` types
- `packages/core/src/i18n/index.ts` — re-export locale types from i18n barrel

- `packages/core/src/types/__tests__/locale.test-d.ts` — type-level tests using `expectType` or `tsd`

**Approach:**

The SDK type generator (Plan 017) emits a typed client per collection. For localized collections, the generator wraps `findMany` and `findOne` in an overload:

```
// Directional — not implementation code
findMany(params: { locale: string; fallback?: boolean } & OtherParams): Promise<LocaleVariant<T>[]>
findMany(params?: OtherParams): Promise<T[]>
```

The overloads ensure that passing `locale` changes the return type to `LocaleVariant<T>`. This is the standard TypeScript overload pattern; the SDK generator emits it as a code string.

`LocaleVariant<T>` is defined as:

```
type LocaleVariant<T> = T & {
  _locale: string
  _localeFallback: boolean
  translatedBy: 'ai' | 'human' | 'pending'
  translationStatus: 'pending' | 'in_progress' | 'complete' | 'error'
  translatedAt: string | null
}
```

The `LocaleQueryParams` type is exported from `@hono-cms/core` for consumers who need to type their own locale-aware fetch calls without the full SDK client.

**Patterns to follow:** Plan 017's SDK generator is the base. The locale extension follows the same per-collection generator pattern — a function that takes `CollectionConfig` and emits additional TypeScript strings.

**Test scenarios (type-level, using `tsd` or `expectType`):**
- `articles.findMany({ locale: 'es' })` infers `Promise<LocaleVariant<ArticlesRow>[]>`
- `articles.findMany()` infers `Promise<ArticlesRow[]>` (no locale fields)
- `articles.findMany({ locale: 'es', fallback: false })` — `fallback` param is `boolean`, not required
- On a non-localized collection (`authors` with `localization: false`): `authors.findMany({ locale: 'es' })` is a TypeScript error
- `articles.translate({ id: '123', targetLocale: 'es' })` exists and returns `Promise<{ status: string }>`
- `articles.getLocales({ id: '123' })` returns `Promise<LocaleStatus[]>`
- `LocaleVariant<ArticlesRow>` includes both the base `ArticlesRow` fields and the locale metadata fields

**Verification:** Type-level tests (`tsd` or `expect-type`) all pass with `bun run typecheck`. Runtime smoke test: SDK client method calls compile without `as any` casts and return correctly typed values from a test endpoint.

## System-Wide Impact

### Database

Every localized collection gains a `{collection}_locale_variants` table. For a CMS with 10 localized collections, that is 10 additional tables. The `articles.id → articles_locale_variants.document_id` FK with `ON DELETE CASCADE` ensures referential integrity. The `UNIQUE(document_id, locale)` index on every locale_variants table ensures upsert correctness and efficient locale lookup.

Existing non-localized collections are completely unaffected — no schema changes, no query changes.

### Build and Bundle

`@ai-sdk/anthropic` and `@ai-sdk/openai` are peer dependencies of `@hono-cms/core`. They are not installed unless the operator configures an AI provider. The `ai` package (Vercel AI SDK core) is a direct dependency — it is always present in the bundle when `@hono-cms/core` is imported. Bundle cost is ~45KB compressed for `ai` alone. The provider SDKs add ~20KB each but are only bundled when imported (dynamic import via `createAIProvider`).

When `config.i18n` is undefined, the i18n module tree is not initialized — middleware is not mounted, routes are not registered. The `ai` package is still in the bundle (it is a static dependency of `@hono-cms/core`). A future optimization could make the `ai` package import conditional on `config.i18n` being defined, but this requires dynamic imports at bootstrap time — defer to v2.

### API Surface

New endpoints added when `config.i18n` is defined:
- `GET /api/{collection}?locale=` — modified existing route
- `GET /api/{collection}/{id}?locale=` — modified existing route
- `POST /api/{collection}/{id}/translate` — new, admin-only
- `GET /api/{collection}/{id}/locales` — new, public (locale status is not sensitive)
- `PATCH /api/{collection}/{id}/locales/{locale}` — new, admin-only (mark as reviewed, direct edit)
- `PUT /api/{collection}/{id}/locales/{locale}` — new, admin-only (direct locale variant edit)
- `POST /cms/jobs/translation` — new, QStash-only (signature-protected)
- `POST /cms/admin/i18n/backfill` — new, admin-only
- `GET /cms/admin/i18n/backfill/status` — new, admin-only

### Admin SPA

The locale switcher integrates into the existing content editor shell (Plan 007). It adds UI complexity to every localized collection's edit view. The settings page adds a new `/settings/i18n` route to the admin router.

### QStash / Jobs

Translation jobs use QStash queues (Plan 010). Each save of a localized document enqueues `(locales.length - 1)` jobs. A site with 4 configured locales and heavy publishing activity could enqueue hundreds of jobs per day. QStash free tier: 500 messages/day. Operators with high publishing volume need a paid QStash tier — document this in the i18n configuration reference.

### RBAC

Two new permission types are introduced:
- `translate:{collection}` — allows calling `POST /{collection}/{id}/translate` (manual trigger). Default: editor and admin roles.
- `i18n:backfill` — allows calling `POST /cms/admin/i18n/backfill`. Default: admin role only.

These permissions integrate with the RBAC middleware from Plan 006.

## Alternatives Considered

### Alternative: Locale column on main table (Option A)

Already rejected in KTD-1. The separate table approach wins on query performance for non-i18n paths, clean schema for mixed localized/non-localized collections, and insert-only backfill mechanics.

### Alternative: Translate eagerly (synchronously) before responding to save

Rejected in KTD-2. 3-6 second response times are not acceptable. Serverless timeout limits (25-30s) make this outright impossible for documents with many locale targets or large rich-text bodies.

### Alternative: Single batch translation prompt (all fields in one call)

Send all localizable fields as a JSON object to the model in a single call, expecting a JSON object back with translated values. This is faster (one API call instead of N) but unreliable: models hallucinate JSON structures, add commentary inside JSON strings, and may reorder keys. Per-field translation with simple string prompts is slower but far more reliable and easier to validate.

### Alternative: Use `ai` package's `createProviderRegistry` for all providers

The Vercel AI SDK's provider registry pattern can centralize all provider config. Rejected for dynamic import conflict: `createProviderRegistry` requires static imports of all providers at module load time, which defeats tree-shaking. The manual dynamic import per provider is more verbose but preserves bundle size guarantees.

## Risk Analysis

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| AI translation quality insufficient for operator's content | Medium | Medium | `translatedBy: 'ai'` flag is always surfaced; editors can correct or retranslate; no auto-publish without human review flag |
| Unexpected AI API cost from high-volume publish activity | Medium | Medium | `autoTranslate: false` opt-out; `translateOnPublish: true` reduces volume; cost estimate shown in admin before backfill |
| QStash message limit exceeded on free tier | Low | Low | Document QStash tier requirements in config reference; admin `/settings/i18n` shows estimated monthly job count |
| Richtext HTML corruption (AI adds/removes tags) | Low | High | Prompt explicitly lists untouchable attributes; post-translation HTML sanitization against original structure can be added as a validation step (deferred to v2) |
| BCP 47 locale codes collide (operator uses 'en-US' and 'en') | Low | Medium | BCP 47 validator warns at config time if two locales share the same base language; fallback chain handles this correctly but may return unexpected results |
| Slate JSON translation breaks rich-text editor state | Low | High | Slate text extraction is a pure transformation tested against reference Slate documents; only `text` nodes are touched; structural nodes are reconstructed unchanged |
| Translation job queue grows unboundedly during outage | Low | Low | QStash retains messages for up to 7 days; no auto-retry (job returns 200 on error); operator can inspect QStash queue in console and manually re-trigger via admin |

## Phased Delivery

### Phase A — Data Model and Config (U1, U2)

Locale variant table schema generation, `I18nConfig` type, BCP 47 validation, locale-aware query builder. No AI calls, no background jobs. Enables testing the schema generation and query logic in isolation before adding the async translation layer.

### Phase B — Translation Engine (U3, U4)

Translation job handler and AI provider factory. At the end of Phase B, `translationJob` can be called directly (in tests and via `curl`) and produce correct locale variant rows. Phase A must be complete.

### Phase C — API Surface and Job Wiring (U5)

Content route locale params, QStash job endpoint, manual translate trigger, locale status endpoint, backfill endpoints. Phase B must be complete.

### Phase D — Admin UI (U6)

Locale switcher component and settings page. Phase C must be complete (UI calls Phase C endpoints). Plan 007 admin SPA foundation must be complete.

### Phase E — SDK Types (U7)

Locale-aware SDK type generation. Plan 017 (SDK generator) must be complete or at least its generator interface must be stable. Can run in parallel with Phase D.

## Deferred Implementation Notes

- The exact Drizzle query syntax for the locale-aware JOIN (COALESCE pattern) should be validated against the D1, Turso, and Postgres adapters during implementation — the SQL is standard but Drizzle's query builder syntax for dynamic COALESCE columns varies by dialect
- QStash queue naming convention (should it be `cms-translation-{collection}` or a single `cms-translation` queue?) is an implementation-time decision based on QStash queue limits and throughput requirements
- Whether the `PATCH /locales/{locale}` endpoint accepts a full locale variant body (for direct editing) or only the `translatedBy` flag update — scope of "Mark as reviewed" vs "Edit translation" actions
- Polling interval (3 seconds) may need tuning based on observed QStash delivery latency in production
